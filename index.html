0.2 Contact vs Subscriber の決定的な違い

ここを混同していると、購読管理・購読数差異・データ構造の問題が全滅する。

概念 定義 識別子 存在場所
Contactチャネル横断の人物レコードContact KeyAll Contacts (Contact Builder)
Subscriber特定チャネルの購読者レコードSubscriber KeyAll Subscribers (Email Studio)
Contact IDシステム内部の数値 ID（ユーザー変更不可）Contact IDバックエンド

重要な含意

  • すべての Subscriber は Contact だが、すべての Contact が Subscriber ではない（メールを送られたことのない Contact もありうる）
  • Sendable DE* から行を削除しても、All Contacts からは消えない（Contact Identity は別レイヤーで管理）
  • ベストプラクティス: Subscriber Key と Contact Key は同じ値にする
  • Email Address を Contact Key にすると、メアド変更で別人扱い・共有メアドで競合などの問題が発生

0.3 Business Unit 構造（Enterprise 2.0*）

試験シナリオの多くが複数 BU* を持つ企業を想定している。

Parent BU (Top-level Account)
├─ Child BU 1 (例: ブランドA)
├─ Child BU 2 (例: ブランドB)
└─ Child BU 3 (例: 部門X)

ポイント

  • All Contacts は Enterprise（全 BU 横断）レベル → Contact 数のカウントも Enterprise 単位
  • All Subscribers は BU ごと（ただし Parent BU からは子 BU を跨いで参照可能）
  • Shared Data Extension は Parent BU で作成し、複数の Child BU から参照可能
  • Send Classification / Sender Profile / Delivery Profile は BU ごとに管理

0.4 試験で出る前提

  • 多くのシナリオが「複数 BU を持つ企業」を想定して出題される
  • 「Parent BU からのみ実施できる作業」「子 BU に反映される設定」「BU を跨ぐデータ共有」の論点は頻出
  • 一斉送信を想定する問題と、API 駆動の 1 件単位送信を想定する問題が混在する

SECTION 1

マクロ：試験全体の構造・配点・攻略方針

1.1 何を測る試験か

Email Specialist は単に「メールを作れるか」を見る試験ではなく、Marketing Cloud Engagement のメール領域で、顧客シナリオに対する設計判断ができるかを問う試験。

具体的には

  • どのデータ構造で購読者・属性・履歴を管理するか
  • どの送信方式（Guided / User-Initiated / Triggered / Journey / Automation Send）を使うか
  • どの自動化ツール（Automation Studio / Journey Builder / Triggered Send）を選ぶか
  • 送信対象が想定より少ない/多い原因を切り分けられるか
  • メール到達性・購読管理・法令対応を踏まえた設計ができるか
  • トラッキング結果を正しく読み、必要なレポート手段を選べるか

1.2 試験仕様（2026年現行）

項目内容
問題数60問（採点対象）+ 最大5問（非採点）
試験時間90分
合格点67%
形式多肢選択・複数選択（multi-select*）混在
受験料$200 USD / ¥30,000 JP（再受験 $100）
受験形態テストセンター or オンライン Proctored
持ち込み不可
Mark for Review*利用可（後で見直しできる）

特筆すべき点

  • 他の Salesforce 試験より 複数選択（"Choose 2" / "Choose 3"）の比率が高い。1問で2-3個正解させる形式に注意。
  • 「best」「only」「all」「which two」「most appropriate」などの条件を限定する語は決定的。
  • 試験画面に下線・ハイライト機能はない。テストセンターではスクラッチペーパー、OnVUE* ではデジタルホワイトボードが提供される。

1.3 配点と優先順位

優先度セクション配点戦略
1Subscriber and Data Management26%最優先。データ設計が崩れると他全部が崩れる
2Marketing Automation26%AS/JB/Triggered の使い分けが頻出
3Content Creation and Delivery24%送信設定・送信方式・コンテンツ管理が広い
4Insights and Analytics14%軽視すると Einstein 系・Data Views で落とす
5Email Marketing Best Practices10%配点低いが、到達性・法令の罠が多い
Data + Automation で 52%、Content まで含めると 76%。ただし複合シナリオが多く、どのセクションも「捨てる」判断は危険。

複合シナリオの例

  • Automation の問題に購読ステータスが絡む
  • Content の問題に Send Classification が絡む
  • Analytics の問題に Data Extract 制限が絡む

1.4 出題思想の変遷

旧バージョンでは Email Message Design や Tracking and Reporting が独立セクションだったが、現行はより実務判断寄りに再編されている。

メール作成
  ↓
データ構造を使ったパーソナライズ
  ↓
適切な送信方式
  ↓
Automation / Journey による運用
  ↓
Tracking / Analytics による改善

近年（Spring '25 / Spring '26 メンテナンスモジュール）の追加論点

  • Journey Builder High-Throughput Sending (HTS*)
  • Einstein 生成 AI による多言語コンテンツ
  • Automation Studio History Dashboard
  • SMS の VerifiedClicks
  • レポートデータ再処理（7日ウィンドウ）
  • 送信・エンゲージメントデータ 180日保持
  • Data Extract ジョブキュー上限 250件
  • Path Optimizer (Journey Builder 内のA/Bテスト)

これらは試験コアを置き換えるものではないが、Analytics / Automation / Journey / Einstein の出題は軽くないことを示すシグナル。

1.5 合格戦略の大枠

「機能名暗記」ではなく「境界判断」の試験であることを意識する。

同じような機能名が並ぶが、問われるのは定義ではなく境界。
